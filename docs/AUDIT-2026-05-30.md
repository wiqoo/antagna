# Antagna-v2 — Prioritized Engineering Report

Audited page-by-page + 4 cross-cutting lenses (permissions, Drive auto-folder, desktop overflow, mobile dashboard). All findings below are verified against current `main`. The single dominant theme: **the DB-layer authz model is correct and the v_*_safe masking layer works, but page-level enforcement is applied inconsistently — many sensitive pages check only `if (!user)` and read base tables directly, bypassing both the read gate and the masking views.**

## CRITICAL — data exposure / unauthorized mutation reachable by any logged-in user

### C1. `/employees` leaks every employee's monthly salary to any authenticated user
- **File:** `apps/web/src/app/employees/page.tsx:31-68` (query) + `EmployeesList.tsx:158-170` (render)
- **Confirmed:** page guards only with `if (!user)`. `can('access.manage')` (line 38) is used *solely* to null the salary client-side — but the SELECT (line 54) pulls `e.monthly_salary` for all employees from the base `employees` table, and the page has no `requirePermission`. Any signed-in account (trainee, videographer, freelancer) lists every employee's salary, job title, email, department, nationality, reports-to.
- **Fix:** Add `await requirePermission('team.read')` after the auth check. Read profiles/employees through `v_team_safe` inside `withProfileScope`, and only select `monthly_salary` when `can('team.read.salaries')`. Conditionally render the salary column on `canViewSalaries`.

### C2. `/inbox/[threadId]` is ungated and reads raw tables (IDOR)
- **File:** `apps/web/src/app/inbox/[threadId]/page.tsx:128-164`
- **Confirmed:** only `if (!user)` (line 128). Queries raw `email_threads` + `email_messages` joined to raw `clients`/`contacts`/`projects`/`profiles` — **not** `v_email_threads_safe`. Any signed-in user (crew/freelancer/HR who see 0 rows in the gated `/inbox` list and hold no `email_threads.read.*` perm) can open ANY thread by UUID and read the full subject, `ai_summary`, every message body, sender emails, client `name_ar`, and primary contact name. There is no `v_email_messages_safe` view, so message bodies have no row protection anywhere.
- **Fix:** `const canRead = await canAny(['email_threads.read.all','email_threads.read.assigned']); if(!canRead) redirect('/dashboard');`. Route the thread query through `v_email_threads_safe` inside `withProfileScope(effectivePid, ...)` exactly like the list page. For messages, first confirm thread visibility via the safe view, then load `email_messages` for that confirmed thread id only.

### C3. WhatsApp send + read pages have zero authz despite `whatsapp.send` existing in the seed
- **File:** `apps/web/src/app/whatsapp/actions.ts:11-60` (send) + `/whatsapp/page.tsx:41` + `/whatsapp/[thread]/page.tsx:45`
- **Confirmed:** `sendWhatsappMessage` does `auth.getUser()` then sends from the company line — no `requirePermissionAction`. The list and thread read pages let any signed-in user read every WhatsApp conversation on the company line. The `whatsapp.send` key is seeded (mig 18 line 52; mig 049) but never enforced anywhere.
- **Fix:** `const actorId = await requirePermissionAction('whatsapp.send')` at the top of `sendWhatsappMessage`; wrap the insert in `withActor(actorId, ...)`. Add a read gate (`canAny(['email_threads.read.all','email_threads.read.assigned'])` or a dedicated comms read perm) → `redirect('/dashboard')` on both read pages.

## HIGH — financial/PII exposure, ungated AI spend, ungated writes

### H1. `/equipment/[id]` leaks purchase price, insurance value, serial — no read gate, no mask
- **File:** `apps/web/src/app/equipment/[id]/page.tsx:70-85` (rendered 245-256)
- **Confirmed:** only `if (!user)` (line 70). Reads base `equipment` directly (lines 79-80: `purchase_price_sar`, `insurance_value_sar`), bypassing the `equipment.read.financial` mask the catalog enforces. Every catalog/reservation/repair/metrics link and the QR scanner land here.
- **Fix:** Add `await requirePermission('equipment.read')`; fetch financial columns through `vEquipmentSafe` inside `withProfileScope(effectivePid, ...)`, mirroring `/equipment/page.tsx`.

### H2. `/projects/[id]/edit` renders unmasked `contracted_value_sar`, no `project.update` page gate
- **File:** `apps/web/src/app/projects/[id]/edit/page.tsx:27-34` (rendered 185-192)
- **Confirmed:** only `getUser`, then SELECTs base `projects` (not `vProjectsSafe`). The write (`updateProject`) is gated, but the form render exposes contracted value to anyone who reaches the URL. Only project page in the domain that bypasses the safe view.
- **Fix:** Add `await requirePermission('project.update')` after the auth check; read the project through `vProjectsSafe`/`withProfileScope` so masking applies on render.

### H3. `/clients/[id]/edit` exposes legal name / VAT / CR to any logged-in user
- **File:** `apps/web/src/app/clients/[id]/edit/page.tsx:21-26,73-74,133-142`
- **Confirmed:** `.select().from(clients)` (full base row) with only `if (!user)`; renders `legalName`/`vatNumber`/`crNumber`. `contacts/[id]/edit` does this correctly at line 28 — this page does not.
- **Fix:** Add `await requirePermission('client.update')`; prefill financial/legal fields only when `can('clients.read.financial')`, else mask/omit. (Also fixes the industry-input inconsistency, see L-tier.)

### H4. `/reports` and `/kpis` leak company financials to any authenticated user
- **File:** `apps/web/src/app/reports/page.tsx:13-88` + `apps/web/src/app/kpis/page.tsx:40-78`
- **Confirmed:** both enforce only `if (!user)`, then aggregate `SUM(contracted_value_sar)` from base projects/clients (reports) and read company-scope `kpi_snapshots` (kpis).
- **Fix:** `await requirePermission('financials.read')` (or `canAny(['financials.read','projects.read.financial'])`) at the top, redirecting non-financial users. Gate financial-scope KPI cards behind `financials.read`.

### H5. `/freelancers` + `/talents` (list & detail) leak rates / commission % / payout refs — and availability writes are ungated
- **Files:** `freelancers/page.tsx:30`, `freelancers/[id]/page.tsx:50`, `talents/page.tsx:25`, `talents/[id]/page.tsx:47` (reads); `freelancers/actions.ts:11-53` (writes)
- **Confirmed:** all four pages have only `if (!user)`; expose `default_rate_sar`, ratings, `email_primary`, `payout_method_ref`, `commission_pct`, `legalName`. No `v_freelancers_safe`/`v_talents_safe` masking views exist. **Worse:** `addFreelancerAvailability`/`removeFreelancerAvailability` (lines 19, 50) only do `if (!user)` then `db.execute` — any signed-in user can add/delete availability windows for any freelancer, with no `withActor` audit.
- **Fix:** Add a page read gate (`requirePermission('team.read')` or new `freelancers.read`/`talents.read` keys) to all four. Add `await requirePermissionAction('access.manage')` (or a scheduling/people write key) to both availability actions and route them through `withActor`. Build `v_freelancers_safe`/`v_talents_safe` to mask rate/commission/payout behind a financial perm.

### H6. AI brief parse on `/projects/new` is not permission-gated (Anthropic token-burn)
- **File:** `apps/web/src/app/projects/new/actions.ts:77-83`
- **Confirmed:** `parseBriefRich` only does `auth.getUser()` (line 82) — no `requirePermissionAction`. The equivalent `parseBrief` in `briefs/new/actions.ts:63` gates on `brief.parse_ai`. Any signed-in user can hit paid Sonnet here.
- **Fix:** `await requirePermissionAction('brief.parse_ai');` at the top of `parseBriefRich`.

### H7. `createTaskFromThread` (WhatsApp → project_task) writes with no permission/assignment check
- **File:** `apps/web/src/app/whatsapp/[thread]/actions.ts:65-110`
- **Confirmed:** only `if (!user)` (line 73), then inserts into `project_tasks` for any project in the all-open-projects dropdown.
- **Fix:** Replace the manual lookup with `const actorId = await requirePermissionAction('project.update')`; wrap in `withActor`; ideally verify the actor is assigned to the chosen `projectId`.

## MEDIUM — incomplete authz, IDOR with UI cover, Drive latency, token-burn

### M1. `setTaskStatus` IDOR — flips ANY project task by id with no ownership scope
- **File:** `apps/web/src/app/tasks/actions.ts:20-28`
- **Confirmed:** the `source==='project'` branch does `.update(projectTasks).set({status,...}).where(eq(projectTasks.id, taskId))` (line 25) gated only by `daily_task.manage_self` (held by every positioned user). The UI only renders the toggle for the actor's own tasks, but the server action trusts the client `taskId` — so any positioned user can flip the status of any project task. The daily branch is correctly owner-scoped (line 32 + `ownerId` insert).
- **Fix:** Restrict the project WHERE to `and(eq(projectTasks.id, taskId), eq(projectTasks.assigneeId, actorId))`, OR gate the project branch on a real project-task write permission and verify project membership. Mirror the daily branch.

### M2. CRM/Project/Brief/Equipment create+edit pages render without page-level guards
- **Files:** `clients/new/page.tsx:20-22`, `contacts/new/page.tsx:22-26`, `projects/new/page.tsx:20-23`, `projects/[id]/board/page.tsx:57-61`, `briefs/new/page.tsx:16-17`, `equipment/{new,kits,metrics,repairs,reservations,scan}/page.tsx`
- **Confirmed pattern:** all check only `getUser`; the corresponding write actions ARE gated, so no write-path leak — but the forms/dropdowns/read data render for users who lack the permission (and `/projects/[id]/board` exposes task titles/assignees to users lacking `project.read`).
- **Fix:** Add the matching guard after the auth check: `requirePermission('client.create')`, `'contact.create'`, `'project.create'`, `'project.read'` (board), `'brief.create'`, `'equipment.read'` (all 6 equipment sub-pages). `contacts/[id]/edit` already does this correctly — bring the rest in line.

### M3. Dashboard MTD revenue / contracted value computed for all users and self-revealable via the customize picker
- **File:** `apps/web/src/app/dashboard/board.tsx:87-95` + `dashboard-grid.tsx:207-225`
- **Confirmed:** `board.tsx` computes `mtd_revenue` (`SUM(contracted_value_sar)`) and per-project contracted value from raw `projects` (no `withProfileScope`, no permission check) for every user, and always builds the `CardMTDRevenue` node into `items[]`. Per-position layouts hide it cosmetically, but the add-card popover lets any user un-hide any hidden-but-live card, and the value is already in the streamed HTML regardless.
- **Fix:** Gate the revenue query + card node server-side behind `can('finance.revenue.read')` so the data is never computed/sent; exclude permission-gated cards from the add-card picker. (Same root issue affects `/my-day`, which embeds the same board.)

### M4. Drive folder is not created at project creation — up to ~2 min latency
- **File:** `apps/web/src/app/projects/new/actions.ts` (`createProject`, ~line 344) — `ensureProjectFolderTree` never called
- **Confirmed:** the full chain exists and is automated via a Trigger.dev cron (`drive-folder-scanner.ts`, `*/2 * * * *` → `/api/integrations/drive/scan`, which selects `WHERE drive_folder_id IS NULL` and writes back). But `createProject` never calls `ensureProjectFolderTree` nor triggers the scanner, so a new project has no Drive folder for up to ~2 minutes and the success path never confirms folder creation.
- **Fix:** After the `withActor` tx returns `newId`, best-effort call `ensureProjectFolderTree({projectCode, projectTitle, clientCode, year})` inside try/catch and UPDATE `drive_folder_id`/`url`. Wrap in try/catch so a Drive outage never blocks project creation (the idempotent cron remains the backstop). Note `createProject` doesn't currently SELECT the generated `code`/`client_code` — add a `RETURNING code` or follow-up select.

### M5. `/social` exposes sponsored-deal contract values
- **File:** `apps/web/src/app/social/page.tsx:30-31,36-37`
- **Confirmed:** only `if (!user)`; reads `sponsoredDeals.contractValueSar` from base tables.
- **Fix:** Gate the page (or at least the deals tab / `contractValueSar` column) behind `projects.read.financial` or a social-specific read perm.

### M6. `/inbox/suggestions` + email-intel refresh gate on legacy admin roles, not comms perms
- **File:** `apps/web/src/app/inbox/suggestions/page.tsx` + `/api/integrations/email-intel/refresh` (both `getAdminUser()`)
- **Confirmed:** the rest of inbox gates on `canAny(email_threads.read.*)`, but suggestions + manual refresh gate on `getAdminUser()` (`system_admin`/`general_manager` only). A `production_director`/PM with comms perms but not in `ADMIN_ROLES` is locked out — inconsistent gate model.
- **Fix:** Replace the admin-role check with the comms permission (`ai_suggestion.approve` / `email_threads.read.all`) so the gate matches the rest of the domain.

## (a) Thin / poor pages needing real content

**None.** Every audited page is genuinely data-wired with real queries and functional actions — there are no stub/placeholder pages. The problems are authz and CSS, not missing content. (`/equipment/scan` and `/equipment/new` have `dataWired=false`, but correctly — they are a client-only camera widget and a create form.)

## (b) Integration / automation gaps

- **Drive auto-folder is poll-based, not synchronous** (see M4). Works and is automated, but ~2 min latency at creation, and if the cron is paused new projects never get folders. **The 10-schedule Trigger.dev Pro cap is fully consumed** (`apps/worker/src/trigger/index.ts`) — adding an 11th `schedules.task` can silently drop the drive scanner's registration. Defend by piggybacking the drive scan onto an existing scanner (as `whatsapp-media-scanner` does) and/or a CI assert that fails if scheduled-task count > 10. Consider widening cron to `*/5` once M4's synchronous path lands.
- **Gmail/Drive depend on a live `google_integrations` row** for `info@voltsaudi.com`. `getDriveClient` throws "No active Google integration" if absent — folder creation silently fails per-project into `report.errors`. **This is the "Gmail-not-connected" failure mode: if the integration row is missing/expired, the inbox sync, email-intel, and Drive folder pipelines all no-op silently.** Surface integration health in the admin UI and make the project-create path tolerant (try/catch) of a missing integration.
- **Leads have no create/detail route** — `/crm` surfaces leads read-only; they only originate from inbound email routing (Pillar 8, not yet wired). Until then the leads section depends on data no UI creates.
- **`client_health_snapshots` (revenue, payment days, trust score) are read but never written** by any CRM route — they render "—" until a background pipeline exists.
- **`contact_methods` supports linkedin/instagram/other but the create/edit forms only collect email/phone/whatsapp** — those types can never be entered through the UI.
- **Duplicate contact-creation entry points** — `clients/[id]/contacts/new` route folders overlap with the inline `addContact` form and top-level `/contacts/new`. Reconcile.

## (c) Desktop horizontal overflow — root cause + fix

**Root cause:** the AppShell content chain clears the fixed RTL right sidebar with **padding only** and has **no overflow containment**. The sidebar is `position:fixed`, `insetInlineStart:0`, `width:248` (`AppShell.tsx:268`). The content column uses `md:ps-[248px]` (`AppShell.tsx:328`) — padding doesn't clip or contain overflow. Neither the column div, nor `<main>` (`:361`), nor the `max-w-[1680px]` wrapper (`:362`) has `overflow-x-clip` or `min-w-0`, and `globals.css` has no document-level guard. So any child wider than the content box overflows and, because the sidebar is painted on top at the visual-right edge in RTL, slides **under** the sidebar and forces a page-wide scrollbar. The data tables themselves are already disciplined (wrapped in `overflow-x-auto`) — the defect is the absent safety net, not any one page.

**Fix (3 small changes, no per-page work):**
1. `AppShell.tsx:328` — change `md:ps-[248px]` → `md:ms-[248px] min-w-0 overflow-x-clip` (margin reserves the gutter; `min-w-0` lets the column shrink; `overflow-x-clip` stops blow-out without creating a scroll container that breaks sticky).
2. `AppShell.tsx:361-362` — add `min-w-0` to `<main>` and to the inner `max-w-[1680px]` wrapper (neutralizes default `min-width:auto` on grid/flex tracks).
3. `globals.css:74` — backstop: `html, body { overflow-x: clip; max-width: 100%; }`.

## (d) Mobile dashboard-card fix

**Root cause:** the live grid is hardcoded to 12 columns at *every* breakpoint (`dashboard-grid.tsx:242` `grid grid-flow-row-dense grid-cols-12`) with no mobile reset, and `cardSpanClass` maps both `sm` and `md` sizes to `col-span-6` at the mobile base (`cards/utils.tsx:14-22`). Since 8 of 10 default cards are sm/md, they render at ~165px (half of a 375px phone) — the "tiny cards." `grid-flow-row-dense` + `items-start` then back-fill and leave stubby empty columns ("shows nothing"). Proof it's an oversight: the Suspense skeleton (`board-section.tsx:26`) correctly uses `grid-cols-1 sm:grid-cols-2`. Same grid powers `/my-day` via `StreamedBoard`, so one fix covers both.

**Fix (2 changes):**
1. `dashboard-grid.tsx:242` — `grid grid-flow-row-dense grid-cols-12 items-start gap-3 md:grid-flow-row` → `grid grid-cols-1 sm:grid-cols-2 md:grid-cols-12 md:grid-flow-row-dense items-start gap-3` (single column on phones, drop dense packing on mobile).
2. `cards/utils.tsx:14-22` — make the mobile base full-width, split only at md+: `sm:'col-span-1 md:col-span-3'`, `md:'col-span-1 md:col-span-4'`, `lg:'col-span-1 md:col-span-6'`, `xl:'col-span-1 md:col-span-8'`, `full:'col-span-1 md:col-span-12'`.

# Antagna-v2 — Product Review: Fonts, WhatsApp gating, AI, IA/Nav, Admin, my-day/dashboard

All six areas were checked against the live code, not just the brief. Bottom line up front: the structural skeleton is good, but four things are genuinely broken and should not ship as-is — **mixed fonts**, **ungated WhatsApp**, **a decorative AI cost-cap**, and **mobile card truncation**. Everything else is polish.

## 1. Fonts — how many, what's inconsistent, the unification plan

**How many: four families are loaded and all four render.** Verified in `apps/web/src/app/layout.tsx`:
- **Geist** → `--font-sans` (Latin UI sans)
- **Geist Mono** → `--font-mono`
- **Vazirmatn** → `--font-arabic-display` (the title font you like)
- **IBM Plex Sans Arabic** → `--font-arabic` (body)

A **fifth** face appears unintentionally: Geist Mono has **zero Arabic / Arabic-Indic glyphs**, so every Arabic string set in `font-mono` silently falls back to an uncontrolled system font. That's the "metadata is in a different font" effect.

**The three real inconsistencies (all confirmed):**
1. **Titles ≠ body.** `globals.css:32` makes body resolve to IBM Plex Sans Arabic; `:33` makes headings resolve to Vazirmatn. Two different Arabic faces on every page, by design.
2. **The good skin is used once.** The `card-clean` skin (`globals.css:257`) that unifies a card onto Vazirmatn body is passed as `variant="clean"` in exactly **one production call site** (`dashboard-grid.tsx:293`) — the other two are `/preview/lab` scratch pages. Every other card renders Arabic titles via the legacy header in `font-mono uppercase`.

## 2. Is WhatsApp gated by permissions? — **No. This is the most serious gap.**
